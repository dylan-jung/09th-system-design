# 제조 설비 이벤트 수집 및 이상 탐지 시스템 설계

> 반도체 증착 공정 모니터링. 초당 수만 건의 센서 데이터를 수집·처리해 이상 징후를 실시간 탐지하는 시스템.

---

## 1. 문제 이해 및 설계 범위 확정

### 시나리오

반도체 Fab 의 증착 장비에서 발생하는 센서 데이터 (온도, 압력, 가스 유량, RF Power 등) 와 운영 로그를 실시간 수집하고, 이상 징후를 탐지해 엔지니어에게 알린다. 설비 제어나 수율 예측 AI 는 범위 밖 — **수집 / 이상 탐지 / 모니터링** 에 집중.

### 도메인 최소 이해

시스템 설계에 필요한 만큼만:

```
Fab (공장) 1개
  └ 장비 (Equipment) 500대
      └ Chamber 1,000개
          └ 센서 50개 (온도/압력/가스유량/RF Power)
              └ 1초마다 값 보고
```

**식별자** (각 이벤트에 붙는 태그):
- `equipmentId` / `chamberId`: 어느 장비의 어느 챔버
- `waferId` / `lotId`: 가공 중인 웨이퍼 / 웨이퍼 묶음
- `recipeId`: 공정 레시피 (설정값 조합)

**이상이란**: 센서 값이 정상 범위를 벗어남 → 누적 시 박막 품질 저하 → 수율 하락. 시스템 관점에서는 "센서가 1초마다 숫자를 보내고, 그 숫자가 이상하면 알림".

### 규모

| 항목 | 수치 |
|------|------|
| 장비 / Chamber | 500대 / 1,000개 |
| 초당 센서 이벤트 | ~50,000 events/sec |
| 일일 이벤트 | ~43억 건 |
| 이상 이벤트 비율 | 0.01~0.1% (정상이 압도적) |
| Dashboard 동시 사용자 | 100~500명 |
| 고해상도 원본 보관 | 7~30일 |
| 집계 데이터 보관 | 1년 이상 |

이 규모 (일 43억 건) 가 모든 설계 결정을 끌고 간다. 일반 웹 서비스보다 수백 배 큰 IoT 스케일.

### 비기능 요구사항

| 항목 | 목표 |
|------|------|
| 수집 지연 | 평균 1초 이내 |
| 이상 탐지 지연 | 평균 3초 이내 |
| 알림 발송 지연 | 이상 감지 후 5초 이내 |
| 데이터 유실 | 중요 이벤트 (알람) 유실 최소화 |
| 가용성 | 설비 운영 시간 동안 지속 |
| 확장성 | 설비/센서 증가에 수평 확장 |

### 핵심 Trade-off (5주차 주제)

이 시스템 설계의 본질은 네 가지 사이의 균형:
- **데이터 유실 ↔ 처리 지연**: 강한 보장 (acks=all + at-least-once) 은 지연/비용 증가
- **알림 정확도 ↔ 응답성**: false positive 줄이면 false negative 증가
- **저장 비용 ↔ 보관 기간**: 고해상도 원본을 오래 보관하면 비용 폭발

## 2. 개략적 설계안

### 핵심 흐름

```
[센서] → [Edge Gateway] → [Kafka] → [Flink 스트림 처리] → 이상 탐지
                                  ↓                          ↓
                          [원본 적재]              [시계열 DB] + [알림]
                          (Object Storage)              ↓
                                                  [Dashboard]
```

### 사용 기술 개념 정리 (학습 메모)

이번 설계에서 쓰는 핵심 기술들:

**Kafka** — 분산 이벤트 스트리밍 플랫폼
- 이벤트를 토픽 (topic) 에 쌓고, 컨슈머들이 가져가는 메시지 버스
- 디스크에 영속 저장 → 컨슈머가 죽어도 offset 부터 재처리 가능
- 파티션 (partition) 으로 분산 → 수평 확장
- 우리 시스템에서: 초당 5만 이벤트의 중앙 버퍼

**Edge Gateway** — 현장 (공장) 의 1차 수집기
- 센서 500대가 Kafka 에 직접 연결하면 연결 폭발 + 관리 불가
- 현장에 Gateway 를 두고: 1차 수집 → 로컬 버퍼링 → 압축/배치 → Kafka 전송
- 네트워크 끊겨도 로컬 버퍼로 데이터 보존

**Flink** — 분산 스트림 처리 엔진
- Kafka 에서 흘러오는 데이터를 실시간 처리 (윈도우 집계, 이상 탐지)
- 핵심 강점: **Event Time 처리** (늦게 도착한 데이터를 발생 시각 기준으로 정확히 처리), 저지연 (ms), 윈도우 + 상태 관리
- **Kafka 와 짝꿍인 이유**: Flink 는 연산 엔진일 뿐 데이터를 저장 안 함 → source 가 필요. Flink 의 장애 복구 (checkpoint) 는 "source 를 과거 지점부터 다시 읽기" 에 의존하는데, Kafka 가 offset 기반 재생을 지원해 이게 가능. 즉 Kafka (재생 가능한 버퍼) + Flink (연산) 는 역할 분담이 맞물린 업계 표준 조합.
- 우리가 Flink 를 고른 이유는 3-2 에서 상술

**시계열 DB** — 시간 기반 데이터에 특화된 저장소
- 센서 데이터처럼 (시각, 값) 쌍이 끝없이 쌓이는 데이터에 최적화
- 시간 범위 쿼리, 다운샘플링, 자동 만료 (retention) 지원
- 후보: InfluxDB, TimescaleDB, Prometheus 등 (3-3 에서 선택)

### 아키텍처

```
        [센서 50,000개]
              │ 1Hz
              ▼
      [Edge Gateway]  ← 현장, 로컬 버퍼링 + 배치
              │
              ▼
      ┌───────────────┐
      │     Kafka     │  토픽: raw-sensor-events
      │  (파티션 분산) │  (equipmentId 기준 파티셔닝)
      └───────┬───────┘
              │
       ┌──────┴──────────────┐
       ▼                     ▼
 [Flink 스트림 처리]   [Sink Connector]
 (이상 탐지)           (원본 그대로 적재)
       │                     │
       │                     ▼
       │              [Object Storage / Data Lake]
       │              (S3, 장기 보관, 7~30일+)
       │
       ├─→ [시계열 DB]      최근 데이터, Dashboard 용
       ├─→ [이상 이벤트 토픽] → [알림 시스템]
       │                          ↓
       │                   [Slack / SMS / Email]
       ▼
 [Dashboard (Grafana)]
```

### 컴포넌트 책임

| 컴포넌트 | 책임 |
|----------|------|
| Edge Gateway | 센서 1차 수집, 로컬 버퍼링, 배치 전송 |
| Kafka | 이벤트 버스, 파티션 분산, offset 기반 재처리 |
| Flink | 실시간 이상 탐지 (윈도우 집계, 임계치/통계) |
| 시계열 DB | 최근 고해상도 데이터, Dashboard 쿼리 |
| Object Storage | 원본 장기 보관 (Data Lake), 사후 분석 |
| 알림 시스템 | 심각도별 발송, 중복 억제 |
| Dashboard | Chamber별 센서 추이 + 이상 이벤트 시각화 |

### 대용량 관점: 병목 지점 분석

초당 5만 / 일 43억 건 규모에서 병목이 생길 수 있는 4개 지점. 상세 설계 (3-1~3-3) 는 각 컴포넌트가 **내부 구조상 왜 이 병목을 견디는지** 를 다룬다.

| # | 병목 지점 | 부하 | 핵심 완화 메커니즘 |
|---|----------|------|-------------------|
| ① | Edge → 중앙 네트워크 | 5만 events/sec × payload | 배치 + 압축 (Gateway) |
| ② | Kafka write | 5만 msg/sec 영속화 | 파티션 분산 + sequential write + pagecache |
| ③ | Flink 처리 | 5만 events/sec 윈도우 연산 | parallelism + 로컬 상태 + 백프레셔 |
| ④ | 시계열 DB write | 5만 point/sec 영속화 | LSM/컬럼 압축 + 배치 write + 다운샘플 |

각 병목은 결국 **"쓰기 throughput 을 어떻게 분산하고 디스크 I/O 를 어떻게 순차화하느냐"** 로 귀결된다. 이게 이 시스템의 CS 적 핵심.

## 3. 상세 설계

핵심 토픽 5개 + 부수 2개:

### 3-1. 대규모 이벤트 수집

#### Edge Gateway 의 역할

500대 설비가 Kafka 에 직접 연결하면 연결 수 폭발 + 네트워크 불안정. 현장에 Edge Gateway 를 두어:

- **1차 수집**: 센서 → Gateway (로컬 네트워크, 안정적)
- **로컬 버퍼링**: 중앙 네트워크 끊겨도 디스크에 버퍼 → 복구 시 전송
- **배치 + 압축**: 1초마다 개별 전송 대신 묶어서 압축 전송 → 네트워크 효율
- **1차 검증**: 스키마 체크, 명백한 garbage 필터

**네트워크 병목 (병목 ①) 완화**: 센서 5만개가 1초마다 개별 패킷을 중앙으로 쏘면 패킷 수 = 5만 pps + 각 패킷의 헤더 오버헤드. Gateway 가 N초/M건 단위로 배치 + 압축하면 패킷 수가 수십~수백 분의 1로 줄고 payload 도 압축됨. 또 센서는 Gateway 와만 통신 (로컬), 중앙과는 Gateway 만 통신 → 중앙 입장에서 연결 수가 5만 → 수십 개 (Gateway 수) 로 격감.

#### Kafka 파티셔닝

```
토픽: raw-sensor-events
파티션 키: equipmentId (또는 chamberId)
파티션 수: 100개 (초당 5만 / 파티션당 500 events/sec 목표)
```

**왜 equipmentId 로 파티셔닝**:
- 같은 장비의 이벤트는 같은 파티션 → 순서 보장
- 이상 탐지가 장비/챔버 단위라 같은 파티션에서 처리하면 효율적

#### Kafka 가 초당 5만 write 를 견디는 구조 (병목 ②)

Kafka 가 대용량 write 에 강한 건 내부 구조 때문:

- **Sequential disk write (append-only log)**: 파티션은 디스크에 순차 append 만 함. 랜덤 write 가 아니라 순차 write 라 HDD/SSD 모두 throughput 이 극대화됨 (디스크는 순차가 랜덤보다 수백 배 빠름). 일반 DB 처럼 B-tree 갱신/랜덤 I/O 가 없음.
- **Page cache + zero-copy**: 쓰기는 OS page cache 에 먼저 들어가고 (즉시 fsync 안 함), 읽기는 `sendfile` 시스템콜로 page cache → 네트워크로 직접 전송 (zero-copy, 유저 공간 복사 생략). JVM heap 을 거치지 않아 GC 부담도 적음.
- **파티션 = 병렬 단위**: 토픽을 100 파티션으로 나누면 write/read 가 100 갈래로 분산. 브로커를 늘리면 파티션을 분배해 수평 확장.
- **배치 + 압축**: producer 가 여러 메시지를 묶어 한 번에 전송 (linger.ms, batch.size). 압축 (snappy/lz4) 으로 네트워크/디스크 절약.

→ 5만 msg/sec 는 Kafka 단일 클러스터엔 여유. 핵심은 파티션 수를 처리량에 맞게 잡는 것 (100 파티션 → 파티션당 500 msg/sec).

#### Hot Partition 대응

특정 장비에 이벤트 몰리면 (예: 고주파 센서 추가) 파티션 불균형:
- 파티션 키를 `equipmentId + chamberId` 로 세분화
- 또는 파티션 수 증설 + 리밸런싱
- 파티션 수는 늘리기만 가능 (줄이기 불가) + 컨슈머 수 ≤ 파티션 수 제약 고려해 초기 설계

#### 유실 vs 지연 우선순위

| 이벤트 종류 | 우선순위 | 설정 |
|------------|---------|------|
| Alarm 이벤트 (중요) | 유실 방지 | `acks=all`, Edge 영속 버퍼 |
| 일반 센서 데이터 (대량) | 지연 최소화 | `acks=1`, 배치 전송 |

`acks=all` 은 모든 ISR (in-sync replica) 이 받아야 ack → 안전하지만 느림. `acks=1` 은 리더만 받으면 ack → 빠르지만 리더 장애 시 유실 가능. 중요 이벤트는 강한 보장, 대량 센서는 throughput 우선. 차등 전략.

### 3-2. 스트림 처리 및 이상 탐지 (Flink)

#### 왜 Flink 인가

| 엔진 | 형태 | Event Time | 지연 | 적합성 |
|------|------|-----------|------|--------|
| Kafka Streams | 라이브러리 | 가능 | 낮음 | 단순 이상 탐지면 충분, 운영 단순 |
| **Flink** | 독립 클러스터 | **매우 강력** | 매우 낮음 (ms) | **본 과제에 최적** |
| Spark Streaming | 독립 클러스터 | 덜 정교 | 중간 (micro-batch) | 이미 Spark 쓰면 고려 |

**Flink 선택 이유**:
1. **Event Time 처리** — 센서 데이터가 Edge 버퍼링/네트워크 지연으로 늦게 도착. "발생 시각" 기준 정확한 윈도우 집계 필요
2. **저지연** — 이상 탐지 3초 이내
3. **윈도우 + 상태** — 이동평균/표준편차 계산에 상태 유지 필요
4. **상태 복구** — 장애 시 checkpoint 부터 재시작 (at-least-once + 멱등 처리로 결과 일관성)

트레이드오프: Flink 는 운영 복잡도 높음. 만약 이상 탐지가 단일 센서 임계치 비교뿐이면 Kafka Streams 로도 충분 (KISS). 본 설계는 통계 기반 (이동평균/표준편차) + 늦게 도착 처리가 필요해 Flink.

#### Flink 가 초당 5만 events 를 견디는 구조 (병목 ③)

- **Parallelism (병렬 서브태스크)**: Flink job 은 여러 operator 로 구성되고, 각 operator 는 N개 병렬 인스턴스로 쪼개짐. Kafka 파티션 100개 → Flink source 병렬도 100 → 파티션별로 독립 처리. parallelism 을 늘리면 throughput 선형 증가 (TaskManager slot 추가).
- **Keyed state + 로컬 상태**: `keyBy(chamberId)` 로 스트림을 키별로 분할하면, 각 키의 상태 (이동평균/표준편차 누적값) 가 특정 서브태스크에 로컬로 유지됨. 상태 접근이 네트워크 없이 로컬 메모리/RocksDB → 빠름. 키가 서브태스크에 분산되므로 상태도 분산됨.
- **상태 백엔드 선택**: 상태가 작으면 heap (빠름), 크면 RocksDB (디스크 spill, 큰 상태 가능). 챔버 1,000개 × 센서 50개의 윈도우 상태는 RocksDB 가 안전.
- **백프레셔 (backpressure)**: 다운스트림 (예: 시계열 DB sink) 이 느리면 Flink 가 자동으로 업스트림 (Kafka source) 의 읽기 속도를 늦춤. 메모리 폭발 없이 자연스럽게 속도 조절. 이게 없으면 느린 sink 때문에 OOM.
- **Credit-based flow control**: 네트워크 버퍼를 credit 으로 관리해 데이터를 받을 여유가 있을 때만 전송 → 백프레셔를 네트워크 레벨에서 정밀 제어.

→ 병목은 보통 **상태 크기** (RocksDB I/O) 또는 **느린 sink** (백프레셔로 전파). parallelism + 상태 백엔드 튜닝으로 대응.

#### Event Time vs Processing Time

```
Event Time:      센서가 값을 측정한 시각 (이벤트에 기록됨)
Processing Time: Flink 가 그 이벤트를 처리한 시각
```

센서 A 가 10:00:00 에 측정한 데이터가 네트워크 지연으로 10:00:05 에 도착할 수 있음.
- Processing Time 기준: 10:00:05 윈도우에 잘못 들어감
- Event Time 기준: 10:00:00 윈도우에 정확히 들어감 ← 우리가 원하는 것

#### Watermark (늦게 도착하는 이벤트 처리)

Flink 의 핵심 개념. "이 시각까지의 데이터는 다 도착했다고 보겠다" 는 기준선.

```
Watermark = 현재 본 최대 Event Time - 허용 지연 (예: 10초)

→ 10초 늦게까지는 기다렸다가 윈도우 닫음
→ 10초 넘게 늦으면 late event 로 별도 처리 (또는 버림)
```

허용 지연을 길게 잡으면 정확도 ↑ 지연 ↑, 짧게 잡으면 반대. Trade-off.

#### 이상 탐지 로직

```
[임계치 기반] — 단순, 빠름
  센서 값이 고정 범위 벗어남 (예: 압력 > 100)

[통계 기반] — 정교
  슬라이딩 윈도우 (최근 1분) 내:
    이동평균 (moving average)
    표준편차 (standard deviation)
  현재 값이 평균 ± 3σ 벗어나면 이상 (3-sigma rule)
```

Flink 의 윈도우 연산으로 챔버별/센서별 통계를 실시간 유지.

#### 이상 탐지 결과 발행

```
이상 탐지됨 → Kafka 토픽 "anomaly-events" 에 발행
  → 알림 시스템이 구독
  → 이상 이벤트 저장소에 기록
```

별도 토픽으로 분리 → 알림과 저장이 독립적으로 소비.

#### 상태 복구

Flink 는 주기적으로 **Checkpoint** (상태 스냅샷) 를 저장 (S3 등):
- 윈도우 집계 상태, Kafka offset 등을 함께 저장
- 장애 시 마지막 checkpoint 부터 재시작
- 재시작하면 일부 이벤트가 재처리될 수 있으나 (at-least-once), 이상 탐지/저장이 멱등하므로 결과는 동일 (effectively-once). end-to-end exactly-once 인프라까지 갈 필요 없음.

### 3-3. 데이터 저장 계층 (Tiered Storage)

일 43억 건을 다 고해상도로 영구 보관하면 비용 폭발. **계층 분리**가 핵심.

| 계층 | 저장소 | 데이터 | 보관 | 용도 |
|------|--------|--------|------|------|
| **Hot** | 시계열 DB | 최근 고해상도 (1초 단위) | 7~30일 | Dashboard 실시간 조회 |
| **Warm** | 시계열 DB (다운샘플) | 집계 (1분/1시간 평균) | 1년+ | 추세 분석 |
| **Cold** | Object Storage (S3) | 원본 전체 (parquet) | 장기 | 사후 수율 분석, 재처리 |

#### 시계열 DB 선택

| DB | 특징 | 적합성 |
|----|------|--------|
| **InfluxDB** | 시계열 전용, 쓰기 빠름, 다운샘플링 내장 | IoT 센서에 적합 |
| **TimescaleDB** | PostgreSQL 확장, SQL 친숙, 관계형 join | 품질 데이터 join 에 유리 |
| Prometheus | 메트릭/모니터링 특화, pull 기반 | 인프라 모니터링용, 고카디널리티 약함 |

**선택**: 센서 데이터 쓰기 부하가 크고 다운샘플링이 핵심이라 **InfluxDB** 가 자연스러움. 단, 품질 데이터 (Metrology) 와 join 분석이 중요하면 **TimescaleDB** (SQL join) 고려.

#### 시계열 DB 가 초당 5만 write 를 견디는 구조 (병목 ④)

일반 RDB (B-tree) 로 초당 5만 insert 하면 랜덤 I/O + 인덱스 갱신으로 금방 한계. 시계열 DB 는 구조가 다름:

- **LSM-tree 기반 write (InfluxDB 의 TSM)**: write 를 메모리 (memtable) 에 모았다가 정렬해서 디스크에 순차 flush. Kafka 와 마찬가지로 **순차 write** 라 throughput 높음. 랜덤 write 인 B-tree 대비 압도적.
- **시간 기반 파티셔닝 (shard/chunk)**: 데이터를 시간 구간별로 나눠 저장. 최근 구간에만 write 집중되고, 오래된 구간은 immutable → 압축/삭제가 구간 단위로 효율적. retention (자동 만료) 도 구간 drop 으로 O(1).
- **컬럼 지향 + 높은 압축률**: 같은 센서의 연속된 값은 비슷함 (예: 온도 25.1, 25.1, 25.2...). 델타 인코딩 + 압축으로 10배 이상 압축. 시계열 데이터의 시간 컬럼은 델타-오브-델타 인코딩으로 극단적 압축.
- **다운샘플링 내장**: continuous query / task 로 1초 → 1분 → 1시간 집계를 자동 수행. 오래된 고해상도는 drop.

**카디널리티 주의**: 시계열 DB 의 최대 적은 **고카디널리티**. `equipmentId(500) × chamberId(1000) × sensorId(50)` 의 조합이 series 수를 결정하는데, 여기에 waferId 같은 무한 증가 태그를 넣으면 series 가 폭발해 메모리/인덱스가 터짐. 태그는 유한 집합만 (waferId 는 field 로, tag 아님).

#### 다운샘플링 (Downsampling)

```
원본: 1초 단위 (Hot, 7~30일)
  ↓ 자동 집계
1분 평균/최대/최소 (Warm, 1년)
  ↓
1시간 평균 (장기 추세)
```

오래된 데이터는 해상도를 낮춰 저장 비용 절감. 최근은 정밀하게, 과거는 거칠게.

#### Hot/Cold 조회 경로 분리

- Dashboard 최근 1시간: 시계열 DB (Hot) — 빠름
- 사후 분석 / 재처리: Object Storage (Cold) — 느리지만 전체 원본

### 3-4. 알림 정확도 및 중복 제어

#### False Positive vs False Negative

```
False Positive (오탐): 정상인데 알림 → 엔지니어 피로, 알림 무시 (양치기 소년)
False Negative (미탐): 이상인데 알림 안 함 → 수율 손실, 진짜 위험
```

제조에서는 미탐 (놓침) 이 더 치명적 (수율 = 돈) → false negative 줄이는 쪽으로 약간 기움. 단 오탐 too much 면 알림 자체를 무시하게 됨 → 균형 필요.

#### 알림 발송 조건

```
[단발 임계치 초과] → 즉시 알림? No (노이즈)
[연속 N회 초과] → 알림 (예: 3초 연속 3회) ← 노이즈 감소
[지속 시간] → 30초 이상 이상 지속 시 알림
```

연속 조건으로 순간 스파이크 (센서 노이즈) 를 거름.

#### 중복 알림 억제

```
같은 챔버, 같은 이상 유형이 반복 → 알림 폭탄
→ Dedup: 일정 시간 (예: 10분) 내 같은 (chamberId, anomaly_type) 은 1회만 알림
→ 상태 변화 시에만 알림: 이상 시작 / 이상 해소 (recovery)
```

#### Recovery 알림

이상 → 정상 회복 시 "해소됨" 알림. 엔지니어가 상황 종료를 인지.

#### 심각도별 채널

| 심각도 | 채널 |
|--------|------|
| Critical (수율 직결) | SMS + 전화 + Slack |
| Warning | Slack |
| Info | Dashboard 만 |

### 3-5. 장애 복구 및 재처리

#### Kafka Offset 기반 재처리

Kafka 는 이벤트를 디스크에 영속 저장. 컨슈머는 어디까지 읽었는지 offset 으로 추적:
- 컨슈머 (Flink) 죽음 → 재시작 시 마지막 commit offset 부터 재처리
- 데이터 유실 없음 (Kafka 에 남아있음)

#### Flink Checkpoint

- 주기적으로 상태 + offset 을 S3 에 스냅샷
- 장애 시 마지막 checkpoint 부터 복구 (at-least-once, 멱등 처리로 결과 일관)

#### 불량 이벤트 처리 (Dead Letter)

```
스키마 오류 / 필수 필드 누락 / 파싱 실패
  → Dead Letter Queue (별도 Kafka 토픽) 로 보냄
  → 정상 파이프라인 막지 않음
  → 나중에 분석 / 수동 처리
```

#### 유실 방지 vs 중복 허용

at-least-once + 멱등 처리가 기본. 이상 탐지는 같은 이벤트 두 번 처리해도 결과 동일하게 (멱등) 설계. 저장은 ClickHouse ReplacingMergeTree 나 시계열 DB 의 동일 timestamp 덮어쓰기로 dedup.

### 3-6. Dashboard (간략)

- **최근 데이터**: 시계열 DB (Hot) 에서 조회 — Grafana 가 InfluxDB 직접 쿼리
- **조회 부하 감소**: Dashboard 는 다운샘플된 데이터 표시 (1초 원본을 다 그리지 않음)
- **이상 이벤트 오버레이**: 센서 그래프 위에 이상 발생 시점 마커 표시
- **Hot/Cold 분리**: 최근은 시계열 DB, 과거 조회는 별도 경로 (Cold)

### 3-7. 품질 데이터 지연 처리 (간략)

품질 검사 (Metrology) 데이터는 센서 데이터보다 **한참 늦게** 도착 (웨이퍼 가공 완료 후 측정).

```
실시간 경로: 센서 → 이상 탐지 → 알림 (즉시)
지연 경로:   품질 데이터 (수 시간~수일 후 도착)
              → waferId / lotId 로 과거 이상 이벤트와 join
              → "이 이상이 실제 수율 저하로 이어졌나" 사후 분석
```

- 실시간 알림과 사후 품질 분석을 **분리** (다른 시스템)
- 연결 키: `waferId`, `lotId`
- 사후 분석을 위해 원본 센서 데이터를 Cold 저장소에 보관 (7~30일+)

## 4. 설계 장점

- **일관된 throughput 원리**: 병목 4지점 (네트워크/Kafka/Flink/시계열 DB) 모두 같은 원리로 해결 — 순차 write (append-only log, LSM-tree), 분산 (파티션, parallelism), 배치 + 압축. 랜덤 I/O 를 순차 I/O 로 바꾸고 부하를 잘게 나누는 게 대용량 처리의 핵심.
- **계층 분리 (Hot/Warm/Cold)**: 최근은 시계열 DB 빠르게, 과거는 다운샘플/Object Storage 로 비용 절감. 일 43억 건을 현실적 비용으로 처리.
- **Flink 의 Event Time 처리**: 늦게 도착하는 센서 데이터를 발생 시각 기준으로 정확히 윈도우 집계. Watermark 로 지연/정확도 균형.
- **유실 vs 지연 차등 전략**: 중요 알람은 강한 보장 (acks=all), 대량 센서는 throughput 우선. 한 가지 정책 강요 안 함.
- **Edge Gateway 버퍼링**: 현장 네트워크 끊겨도 데이터 보존. 복구 시 재전송.
- **장애 복구**: Kafka offset + Flink checkpoint 로 at-least-once. 컨슈머 죽어도 과거 시점부터 재처리, 멱등 처리로 중복 영향 없음 (effectively-once).
- **실시간/사후 분리**: 즉시 알림 (실시간) 과 품질 연계 분석 (배치) 을 독립적으로 운영.

## 5. 설계 단점 / Trade-off

5주차 핵심 주제인 네 가지 trade-off 정리:

- **데이터 유실 ↔ 처리 지연**: acks=all + at-least-once 는 안전하지만 지연/throughput 비용. 그래서 대량 센서는 acks=1 로 약간의 유실 감수. 완벽한 무유실은 불가능 (Edge ~ 중앙 네트워크 구간).
- **알림 정확도 ↔ 응답성**: 연속 N회 조건으로 오탐 줄이면 그만큼 탐지가 늦어짐 (false negative 위험). 제조 특성상 미탐이 더 치명적이라 균형점이 까다로움.
- **저장 비용 ↔ 보관 기간**: 고해상도 원본을 오래 보관하면 사후 분석에 좋지만 비용 폭발. 다운샘플링으로 절충하지만 과거 고해상도 데이터는 잃음 (재처리 불가 구간 발생).
- **Flink 운영 복잡도**: 강력하지만 클러스터 운영 (JobManager, TaskManager, checkpoint 저장소) 부담. 단순 임계치 탐지뿐이면 Kafka Streams 가 더 적절. 본 설계는 통계 기반 + late event 때문에 Flink 선택.
- **시계열 DB 카디널리티**: equipmentId × chamberId × sensorId 조합이 많으면 (고카디널리티) 시계열 DB 부하. 태그 설계 주의 필요.
- **exactly-once 대신 at-least-once + 멱등 선택**: end-to-end exactly-once 는 구성 복잡 (two-phase commit, transactional sink) + 지연/처리량 비용. 본 시스템은 이상 탐지가 deterministic, 시계열 DB 저장이 동일 timestamp 덮어쓰기, 알림이 dedup 이라 모든 단계가 멱등 → at-least-once + 멱등 = effectively-once 로 충분. 단 이 선택은 모든 sink 가 멱등하게 설계됨을 전제하므로, 멱등하게 만들기 어려운 sink 추가 시 재검토 필요.
