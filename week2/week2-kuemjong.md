**시나리오**

```
승객이 호출하면 주변 빈 택시를 찾아 매칭하고, 매칭된 택시가 픽업 지점에 도착할 때까지 승객은 택시 위치와 도착 예상 시간을 실시간으로 확인한다.
픽업 이후 목적지에 도착할 때까지 위치 추적이 이어진다.

그외 시나리오는 자유롭게 구체화해도 좋다. 자율주행택시, 빵택시, AI택시 등 OK. 브랜드 이름 지어도 됩니다 (본인 이름 등)
```

**클클 택시**

- 모바일 앱만 지원한다.
- 단일 대도시권에 대한 설계만 수행한다.
    - 도시 권역 내의 행정구역은 구분되어 있다.
- 가입 승객 수는 얼마나 되는가?
    - DAU 200,000 명
    - MAU 800,000명
- 승객의 호출량은 어떻게 되는가?
    - 일 단위 호출 수는 평균 500,000 건
    - 피크 시간대 호출 집중도: 평균 대비 5배 이상
        - 평일 출근(07:30–09:30)
        - 평일 퇴근(18:00–20:00)
        - 금/토 심야(23:00–02:00)
    - 월 초(1일)의 사용량이 월 말(30일)의 사용량 대비 1.5배 높으며, 중순(15일) 이후의 사용량의 급감이 관측된다.
- 하루 동안 가용할 수 있는 기사의 총량은 얼마인가?
    - 누적 가입 기사: 약 50,000명
    - 동시간대에 평균적으로 10,000명의 기사가 운행한다.
    - 하루 누적으로 20,000 명의 기사가 운행한다. (중복 제외)
- 교통량 정보를 확인할 수 있는가?
    - 외부 도로망 서비스를 통해서 10분 단위의 수치를 확인할 수 있다.
- 고객(승객, 기사) 애플리케이션의 구현에 대해서는 하지 않는다.
    - 통신 방법을 정의하는 수준으로 끝낸다.

**2. 개략적 설계안 제시 및 동의 구하기**

**핵심 흐름 (필수)**

1. 기사는 운행 가능 상태가 되면 MQTT로 서버에 연결하고, N초 단위로 현재 위치를 **기사 위치 갱신 API**에 전송한다.
2. **기사 위치 갱신 API**는 스키마나 timestamp 정도만 검증한 뒤 Kafka broker 향으로 topic-publish한다. (위치 및 상태 정보)
3. **기사 위치 갱신 Consumer**는 Kafka 이벤트를 컨슘하여 Redis GEO/driver state를 갱신한다.
    - 이때 기사 상태가 운행 가능한 경우에만 검색 인덱스에 포함한다.
4. 승객이 호출하면 **탑승 요청 API**가 탑승 요청 request (match)를 생성하고, Matcher가 Redis geo index에서 반경 내 빈 기사 후보를 찾는다.
5. Matcher는 직선거리 기준으로 1차 후보를 줄이고, Routing Adapter를 통해 ETA/도로 거리 정보를 가져와 후보 순위를 보정한다.
6. Matcher는 상위 후보 기사에게 제안을 보낸다.
   - 제안 전에는 짧은 TTL lock을 잡아 같은 기사가 여러 호출에 동시에 노출되는 일을 줄인다.
7. Notifier는 MQTT broker/gateway를 통해 기사 앱에 탑승 요청 탑승 제안를 전달한다. 응답이 없으면 push notification을 보조로 사용하거나 탑승 제안를 만료시킨다.
8. 기사가 수락하면 서버가 transaction으로 탑승 요청 상태와 driver 상태를 원자적으로 변경한다. 동시에 수락이 들어오면 첫 번째 성공 요청만 매칭된다.
   - 매칭 트랜잭션이 시작되면 일차적으로 선정된 기사의 상태를 **Matcher** 를 통해 반영한다.
   - 최종 매칭이 결정되면 **Matcher**를 통해 탑승 DB에 최종 매칭 상태를 반영한다.
9. 매칭 이후에는 기사 위치 업데이트가 계속 Redis에 반영되고, 승객 앱은 **탑승/운행 이벤트 Handler**을 통해 기사 위치, ETA, 상태 변경을 받는다.

크게 3가지의 책임 영역을 둔다.
- 각 책임 영역의 장애는 다른 영역에 주는 영향이 최소화 된다.
- 기사 위치 갱신에 대한 책임 (빨간색)

    - 핵심 컴포넌트: Matcher
- 기사/승객 앱과의 이벤트에 대한 책임 (초록색)
    - 매칭 코어 로직 직전/후에서의 매칭 흐름 생명주기 관리 (시작과 끝) 



> 아래 8가지 질문은 답변 **선택**이지만, 본인이 짚고 넘어가는 부분을 1~2개 골라 깊이 있게 다루는 것을 권장.

**3-1. 기사 위치 업데이트 주기**
- 기본 원칙은 "후보 검색용 위치"와 "승객에게 보여주는 위치"의 신선도 요구가 다르다는 것이다.
  - 하기 수치는 경험적/정책적으로 결정되었다고 가정한다.
- 대기 중 : 10초 주기로 가정한다면
    - 동시 운행 기사 10,000명 기준 약 1,000 events/sec 수준
    - 갱신 주기를 10초보다 크게 가져가도 무관할 것으로 보임.
- 제안 수신/매칭 중: 5 ~ 10초 이내 주기.
    - 탑승 제안 수락 직후 위치 갱신 주기를 조금 더 fresh하게 가져감.
- 픽업 이동 중: 5초 주기
    - 실제 승객과 1대1 매칭된 기사이므로 위치를 자주 갱신해 승객이 기사 접근 상황을 보고 있으므로 체감 지연을 낮춘다.
- 운행 중: N초 주기
    - 승객 위치 추적과 ETA 갱신은 필요하지만, 조금 더 stale 해도 되는 부분
    - 승객은 택시 앱 이외의 지도 앱 등 다른 앱들 통해서 현재 위치를 볼 수 있기에 현 위치 갱신 속도가 주는 품질 체감이 상대적으로 크지 않을 수 있다.
- 앱이 일정 시간 동안 위치를 보내지 않으면 `stale`로 판단한다.
    - 대기 중 상태에서는 30초 이상 미수신 시 후보 검색에서 제외한다.
    - 60초 이상 미수신 시 offline에 가깝게 처리한다.

**3-2. 기사 위치는 어떻게 관리할 것인가?**
- Cell 단위 후보군으로 현재 지역의 후보 택시(기사)를 H3로 관리
    - 경계 처리, neighbor 처리 ...
    - cell 단위에서 가용할 수 있는 기사 ID 를 갱신 -> 동시 운행 기사 1만명 기준 1000 RPS 정도 수준
- 우리가 유지하고 싶은 것은 "최신 상태"의 기사 위치, 위치 변동에 대한 이력은 필요 없다.
    - Redis와 같은 인메모리 저장소에 read/write로 1차 임시 저장소 설정
    - 단, 인메모리 저장소 단일 구조에 의존하면 장애가 전체 시스템에 영향을 미친다.
    - 우리가 기대하는 것은 단일 저장소 장애는 "품질 저하(요청 레이턴시 증가, 간헐적 실패 등)"에 국한되게 해야 한다.
    - OLTP 를 fallback으로 두어 장애 포인트에 대한 강건성을 어느정도 향상시킬 수 있다.
- Redis에는 두 종류의 데이터를 둔다.
    - 검색 가능한 기사 위치 인덱스
    - 기사 ID 별 데이터: 위-경도, 정확도, 기사 상태, 마지막 갱신 시점
- 쓰기 경로는 기사 위치 갱신 API -> Kafka -> Consumer -> 저장소 로 가져간다.
    - 기사 위치 갱신 API가 Redis에 직접 쓰지 않기 때문에 Redis 지연이 기사 위치 수신 API에 바로 전파되지 않는다.
    - 같은 기사의 이벤트가 같은 partition에 들어가 순서가 최대한 보존되게 한다.
    - 메세지는 잠깐 유실되어도 괜찮다. 갱신이 일시적으로 지연되어도 시스템 동작에 영향은 적다.
- 조회 경로는 Matcher가 Redis를 직접 조회한다.
    - 매칭 시점의 빠른 반경 검색이 중요하므로 RDB/PostGIS보다 인메모리 geo index를 우선한다.

**3-3. 검색 반경과 결과 처리**
- 기사 후보군이 부족하면 반경을 단계적으로 확장한다.
    - 500km -> 1km -> 3km -> ...
- 반경 검색 결과가 택시 기사 후보군이 너무 많다면?
    - 무작위 기사 N 명에게만 매칭 알림을 보내거나
    - 직선거리 기준 상위 N명만 남기거나
    - ...
- 결과가 없으면 즉시 실패시키지 않고 짧은 대기열 상태를 둘 수 있다.
    - 명시적으로 큐 시스템 등을 운영할 필요는 없다.
    - 사용자 Action 혹은 일정 시간 후 사라지는 팝업/모달 등으로 해결
- 전체 매칭 시간 N분을 넘기면 실패 처리한다.
    - 그 전까지는 반경 확장, 탑승 제안 windows/batch 확대, 인접 cell 검색을 시도한다.
- 단순 거리보다 수락 가능성을 함께 고려한다.
    - 너무 먼 기사에게만 순차 제안하면 승객 대기 시간이 길어진다.
    - 기사에게 주는 인센티브 시스템을 활용할 수 있다.

**3-4. 위치 / 경로 / ETA 전달 / 갱신 방식**

- 기사의 현 위치 업데이트는 HTTP API로 이벤트 형태로 수신한다. 
    - 위치 이벤트는 N초 단위이고 (e.g. 10초), 최신 위치만 중요하기에 멱등적으로 관리한다.
    - 시스템이 관리하는 '위치'는 연속적일 필요 없고, 갱신이 이루지지 않은 stale과 최신 상태인 fresh만 존재하면 된다.
- 기사 애플리케이션 <-> 서버 간의 연결은 MQTT를 사용한다.
    - vs Websocket: 결국 WS 사용으로 인해 직접 구현해야 하는 로직이 많아짐 (재전송, 재연결, 이외의 예외처리 등...)
- 승객에게 전달하는 이벤트는 기사 위치, ETA, 운행 상태 변경, 탑승 제안/matching 결과 등이다.
- 외부 지도 SDK는 승객 앱이 직접 사용하여 지도 격자 렌더링을 처리한다.
    - 계산된 경로는 서버가 Routing Adapter를 통해 계산한 값을 내려준다.

**3-5. 매칭 대상 선정 전략**
- 1차 후보는 H3 cell index의 직선거리 기준으로 빠르게 찾는다.
- 2차 정렬은 Routing Adapter의 ETA를 사용한다. 실제 도로망에서는 직선거리보다 ETA가 승객 경험에 더 가깝다.
- 후보 선정 점수는 ETA, 직선거리, 기사 상태, 수락률, 최근 취소율, 차량 타입 등을 조합할 수 있다.
- 일반 상황에서는 순차 또는 작은 Window 제안을 사용한다. (e.g. 상위 N명에게 제안하고 일정 시간 내 응답이 없으면 다음 슬라이드/배치로 넘어간다.)
- 피크 시간대 또는 수락률이 낮은 지역에서는 3~5명 동시 제안을 허용한다.

**3-6. 이상 상황 판단 / 처리**
- 기사 앱이 위치를 보내지 않으면 상태별 timeout으로 판단한다.
    - 대기 중: 30초 이상 위치 미수신 시 후보 제외, 60초 이상이면 offline 처리
    - 픽업 이동 중: 15~30초 이상 미수신 시 승객 화면에 위치 갱신 지연 표시, 계속 미수신 시 재매칭 정책을 적용할 수 있다.
    - 운행 중: 운행 상태는 유지하되 위치 신뢰도를 낮추고 재연결을 기다린다
- 애플리케이션에 대한 연결 성은 하트 비트를 기준으로 fresh/stale/offline으로 본다.
- 탑승 제안에 대한 무응답은 거절과 비슷하게 처리하되 기사 품질 지표에는 별도 카운트로 남긴다.
- 승객이 호출을 취소하면 탑승 요청 상태를 취소로 바꾸고, 진행 중인 탑승 제안와 driver lock을 해제하며 기사 앱에 cancel 이벤트를 보낸다.
- 네트워크 단절 후 재연결에 대한 정책 처리는?
    - 재연결 시점에 탑승 DB 상태를 기준으로 현재 탑승 요청/제안 상태를 다시 내려준다.

**3-7. 외부 지도 / 경로 서비스 의존 관리**

- 앱의 지도 view 렌더링은 지도 SDK가 직접 처리한다. 
- 지도 렌더링 외 경로 계산을 위한 의존도는 분리한다. (Routing Adapter)
    - Matcher를 통해서 후보 기사 상위 N명에 대해 batch로 ETA를 요청한다.
- 픽업 이동 중에는 기사 위치가 갱신될 때마다 매번 외부 routing을 호출하지 않는다.
    - 위치 변화가 크거나 일정 주기가 지났을 때만 ETA를 재계산한다.
- 외부 지도 view 장애는 다음과 분리한다.
    - 매칭을 완전히 중단하지 않는다. 정확한 ETA 정렬은 약해지더라도 Redis geo + 직선거리 + 평균 속도 기반 추정으로 가까운 기사에게 제안을 계속 보낸다.
    - 운행 과정에서의 경로 시스템 장애에도 최초로 발급된 경로 혹은 마지막으로 갱신된 경로를 stale로써 유지한다.
      
**3-8. 수요 폭주 예측 시 대응 *(예: 잠실 콘서트 종료 30분 내 2만 건 호출)***

- 기본적으로 정책에 기반해 사전 정의된 desired state를 사용한다.
    - 시/공간적 특성에 따른 데이터가 수집되고 있다고 가정하고, 시간에 따른 정적인 desired-state를 정적으로 운영한다.
    - 이벤트 종료 시간과 장소를 알고 있다면, 동일하게 관련 컴포넌트를 사전 증설한다.
      - 해당 cell에서의 탑승 호출은 기사 후보군 검색 cell을 공격적으로 늘려갈 수 있다.   
    - 오토 스케일링 적용 여부의 경우, workload가 스케일인 되면서 기존 요청을 graceful shutdown 할 수 있는가에 초점을 둔다.
- 기사 앱에는 이벤트 지역 진입 인센티브를 사전에 노출해 가용 기사 수를 늘린다.
    - 기술적인 방법이 아닌 방향으로 접근하면, 고비용 택시 호출 기능 (블루, 벤티...) 등에 대한 가중치를 둘 수 있다.
- 호출 피크 시에는 모든 요청을 동시에 매칭하지 않고 짧은 대기열을 둔다. 승객에게는 예상 대기 상태를 명확히 보여준다.
- Cell 확장과 동시 탑승 제안 windows/batch 크기를 평시보다 공격적으로 조정한다.

---

**4. 설계 장점**

- 기사 위치 수집 흐름이 Kafka로 비동기화되어 순간적인 위치 갱신 요청 피크에도 대응할 수 있다.
- Receiver가 얇기 때문에 API 응답이 빠르고, 위치 처리 로직을 consumer 쪽에서 독립적으로 확장할 수 있다.
- 기사에게의 알림은 MQTT 기반 처리로 별도 WS에 대한 로직 구현이 필요 없다.
- 매칭 확정은 서버 transaction으로 처리하므로 동시 수락 상황에서도 한 명만 배정된다.
    - 최종 SoT를 통해 멱등적으로 관리함
- 위치 인덱스, 매칭 상태, 실시간 연결 등 책임 영역 분리를 통해 독립적으로 관리 및 확장이 가능하다.
- 정책적으로 피크 시점에 스케일 아웃을 관리하여, 예측할 수 있는 운영이 가능하다. 

---

**5. 설계 단점**

- Kafka, Redis, MQTT, Routing Adapter 등 컴포넌트가 많아져 초기 구현과 운영 난이도가 높다.
- 데이터 관리 위치가 분산되어서 매칭이나 기사 상태 등에 대한 정합성 관리의 복잡도가 올라간다.
    - 특히 기사 상태 전이에 대한 경합을 신경써야 한다.
    - 통신 홉이 많아지고 트랜잭션이 보수적으로 Commit/Revert 될 수 있어야 한다.
